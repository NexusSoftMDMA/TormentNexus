## Description

Cosmetic bug deferred from the #120 B4 integration audit. After per-connection profile switching (Design B) landed, the Skills tab "What's New" pill shows a bogus "N new / M changed" diff because the snapshot baseline is keyed by a FIXED UUID (iOS sharedContextID ...A1) / serverID-only, not by the (server, profile) pair. Each Hermes profile has its own skills dir (HERMES_HOME/skills), so switching profiles diffs the new profile's skills against the old profile's baseline. Also bleeds across servers on iOS. Skills LIST is correct; only the diff pill is wrong.

## Plan

Fix ONLY the snapshot baseline key; leave the skills list + SSH connection pooling untouched.

1. HermesProfileScope.profileName(forHome:) — new pure helper, inverse of resolveHome: <root>/profiles/<name> -> normalized name; root home -> nil.
2. SkillSnapshotService — optional profile: String? (normalized in init; nil/default/invalid -> bare per-server key = pre-#120 key, no migration). static storageKey(for:scope:). Thread scope through diff/markSeen.
3. SnapshotBackend enum + Mac/iOS/InMemory backends — additive scope: String? = nil; compose key via storageKey. InMemory now String-keyed.
4. iOS SkillsView — read @Environment(\.serverContext) (real serverID + effective profile home injected by ScarfGoTabRoot); snapshot uses serverContext.id + profileName(forHome: paths.home). VM stays on fixed A1 (pooling unchanged). Orphaned A1 baseline -> previousSnapshotEmpty primes silently (no false pill on upgrade).
5. Mac SkillsView — same derivation (nil on local root = unchanged key; correct if remoteHome points at a profile dir).

Tests (ScarfCore swift test, Swift Testing, Linux-safe): profileName cases + round-trip in HermesProfileScopeTests; new SkillSnapshotServiceTests via InMemorySnapshotBackend (independent profile baselines, no false diff on switch, switch-back preserved, cross-server isolation, default/invalid normalization, real-delta detection, storageKey contract). Build scarf mobile for iOS 26.x sim.
