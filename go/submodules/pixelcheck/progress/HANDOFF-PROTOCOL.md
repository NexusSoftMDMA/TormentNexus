# 多对话执行协议（Handoff Protocol）

> **目的**：让 PixelCheck 47 项升级可以**分阶段、跨对话**执行，避免单一对话过长 / 上下文过载
> **核心思想**：状态保存在文件，每个对话有标准启动提示
> **状态文件**：[STATUS.md](./STATUS.md)
> **方案文件**：[../PixelCheck-实施方案与执行计划.md](../PixelCheck-实施方案与执行计划.md)

---

## 启动新对话的标准提示（直接复制）

### 通用启动提示（推荐，最简单）

只需要在新对话里发送这一句：

```
继续 PixelCheck 升级实施
```

Claude 会自动：
1. 读取 `~/Developer/OpenTools/progress/STATUS.md`
2. 读取 `~/Developer/OpenTools/PixelCheck-实施方案与执行计划.md`
3. 报告当前状态 + 提议本次会话该做什么
4. 等你确认后开始执行

### 指定具体任务的启动提示

如果你已经知道这次想做哪个任务：

```
继续 PixelCheck 升级实施。本次任务：M1-1
```

或者多个任务合并：

```
继续 PixelCheck 升级实施。本次任务：M7-1 + M7-2 + M7-3 + M7-6（合并 Wave C 的 M7 体验项）
```

### 检查状态提示（不执行只查询）

```
看 STATUS.md 告诉我当前进度
```

### 切换任务（同一对话内）

如果一个对话内做多个任务：

```
切换到 M{N}-{X}
```

## 每个执行对话的标准流程

每次新对话 Claude 应当按这个流程：

### 1. 启动阶段（前 2-3 分钟）
- 读取 `STATUS.md` 确认当前位置
- 读取 `PixelCheck-实施方案与执行计划.md` 对应任务的详细规格
- 简要 sanity check 项目代码当前状态（git status / 上次 commit）
- **报告当前理解 + 本次要做的具体内容 + 等用户确认**

### 2. 执行阶段
- 按方案执行
- TodoWrite 跟踪本对话内的子任务
- 每完成一个子任务立即 commit（提供 commit hash）
- 遇到偏离方案的决策需用户确认

### 3. 验证阶段
- 跑测试
- 确认 DoD 10 项满足
- 文档更新

### 4. 收尾阶段（必做）
- 更新 `STATUS.md`：
  - 完成项打勾 + 加 commit/PR 链接
  - 进行中项标进度 %
  - 本次决策记录到 "关键决策记录"
  - 更新 "上次会话摘要"
  - 更新 "下次会话起点" 的建议
- 在对话结尾给出**下次会话启动提示**（标准格式）

### 5. 给用户的最终输出

最后一句必须是这种格式：

```
---
## 下次对话启动提示

复制下面这句到新对话：

`继续 PixelCheck 升级实施。本次任务：M{N}-{X}`

或者使用通用提示：

`继续 PixelCheck 升级实施`
（Claude 会自动读 STATUS.md 提议下一个任务）
```

## 状态文件维护规则

### STATUS.md 必须每次会话末尾更新

字段：

- `当前 Wave` / `当前任务` / `状态` / `完成度`
- 任务 checkbox（已完成 → `[x]` + commit ref）
- `当前进行中任务详情`（如果有跨会话进行中的）
- `阻塞项`
- `关键决策记录`（执行中新增的决策）
- `上次会话摘要`（关键改动 / 完成度 / 遗留问题）
- `下次会话起点`（推荐 + 备选）

### 不要更新的内容

以下内容永远从主方案文件读取，不复制到 STATUS.md：

- 任务详细规格（在 PixelCheck-实施方案与执行计划.md）
- ADR（在主方案附录 D）
- 风险登记册（在主方案第 7 章）

## 跨会话上下文管理

### 持久化的（在文件里）
- ✅ 任务进度（STATUS.md）
- ✅ 已完成任务（git log + STATUS.md）
- ✅ 决策（ADR + STATUS.md "关键决策记录"）
- ✅ 代码改动（git）
- ✅ 测试结果（git + CI）

### 不持久化的（每次会话重新建立）
- 当前思路 / 在思考的方向
- 临时调试信息
- 短期 trade-off 讨论

**原则**：任何对未来重要的内容必须落到文件，不能依赖对话记忆。

## 常见问题

### Q1：忘了上次做到哪了
直接说 `继续 PixelCheck 升级实施`，Claude 会自动读 STATUS.md。

### Q2：想跳过某个任务
告诉 Claude：`跳过 M{N}-{X}，直接做 M{Y}-{Z}`
Claude 会评估依赖并提示风险。

### Q3：发现某项做错了想重做
告诉 Claude：`回退 M{N}-{X}，重新评估`
Claude 会找到对应 commit + 提议方案。

### Q4：一个对话执行了多个任务
正常。STATUS.md 都更新即可。

### Q5：执行中发现需要 ADR
Claude 在对应 commit 一并加 `docs/decisions/ADR-XXX-{slug}.md`，并在 STATUS.md 的"关键决策记录"提及。

### Q6：测试失败 / 卡住
Claude 应当：
- 不强行 hack 通过
- 在 STATUS.md 标记 "阻塞项"
- 给出 root cause 分析
- 等用户决定（修代码 / 调整方案 / 跳过）

### Q7：方案写的细节和实际代码不符
按"代码实际状态"为准，更新方案文件标注。Claude 必须明确说"方案 vs 实际"差异。

## 多对话执行的优势

1. **避免单一对话过长**：每个对话聚焦 1-2 个相关任务
2. **状态明确**：随时能从 STATUS.md 看到进度
3. **可暂停可恢复**：今天做不完，明天接着来不丢上下文
4. **错误隔离**：某次会话出错只影响那一段，不污染整个项目
5. **并行潜力**：未来如有多个执行者，按 STATUS.md 分工
