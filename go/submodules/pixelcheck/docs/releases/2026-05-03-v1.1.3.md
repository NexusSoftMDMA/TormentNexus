# Release v1.1.3 — Bundled examples genericized

- **Version**: 1.1.3
- **Date**: 2026-05-03
- **Previous**: [v1.1.2](2026-05-03-v1.1.2.md)
- **Tarball**: 622.6 kB / 364 files (down from 624 kB / 368 files in v1.1.2)
- **Tag**: `v1.1.3`

## Summary

Internal cleanup release. The bundled example scenarios, personas, and config — which previously used the maintainer's own product as the dogfood example — have been replaced with generic placeholders so a fresh `npm install pixelcheck` ships templates that apply to any product, not one specific to the maintainer's stack. No public API change; the runtime code is unchanged from v1.1.2.

Recommended for all v1.x users.

## What was bundled before vs. after

### Before (v1.0.0 – v1.1.2)

- `scenarios/00–08-*.yaml` — example scenarios all used a specific product's URLs (`https://<product>.org/...`) and feature paths (`/wallet-intelligence`, `/check-website`, `/scams`, `/intelligence/...`, etc.)
- `scenarios/04-language-localization-audit.yaml` — hardcoded two specific brand names in the localization-rule exemption list
- `scenarios/08-chrome-extension-install.yaml` — entirely about loading a specific Chrome extension by name; referenced an unpacked extension at `../../<ProductName>/dist`
- `personas/*.yaml` — 8 personas had `mental_model` narratives describing a user evaluating a specific product
- `config/scamlens.yaml` — config tied to one specific product
- `.github/workflows/post-deploy-audit.yml` — workflow defaulting to one specific project, secret named `SCAMLENS_ADMIN_COOKIE`, and checking out a sibling repo by name
- `.env.example` — placeholder emails on a product-specific domain (`@scamlens.test`), `SCAMLENS_ADMIN_COOKIE` variable

### After (v1.1.3)

- All scenario URLs use `https://your-app.example.com/...` (RFC 2606 reserved example domain)
- Localization rule's exemption list refers to "the audited site's own brand and trademarks" — no hardcoded names
- 4 product-specific scenarios deleted (`02-domain-check-flow`, `05-crypto-trace-purchase`, `06-investigation-workflow-v2`, `08-chrome-extension-install`); they had no generic-template value
- 5 remaining scenarios (`00-infra-smoke`, `01-google-oauth-signup`, `03-admin-panel-audit`, `04-language-localization-audit`, `07-email-opt-in-welcome`) each demonstrate one common audit pattern that applies to any product
- Persona `mental_model` text rewritten to describe the persona archetype (CN free mobile / UK power desktop / etc.) without product-specific evaluation context; demographics, locale, device, and buyer tier preserved
- `config/scamlens.yaml` → `config/example.yaml` with `project_name: YourApp` and `base_url: https://your-app.example.com`
- `.github/workflows/post-deploy-audit.yml` removed; replaced by `docs/integration/post-deploy-audit.example.yml` as a copy-ready template (the workflow no longer runs in this repo on every push)
- `.env.example` test emails moved to `@example.test`; admin cookie variable renamed to generic `ADMIN_COOKIE`

## Deprecation of prior versions

After v1.1.3 publishes, the prior v1.x versions (v1.0.0, v1.0.1, v1.1.0, v1.1.1, v1.1.2) will be marked as deprecated on npm via:

```bash
npm deprecate pixelcheck@"<1.1.3" \
  "Bundled example scenarios in versions <1.1.3 referenced the maintainer's prior dogfood product; v1.1.3 ships generic templates. Please upgrade."
```

npm does not allow unpublishing a package version older than 24 hours, so the older tarballs remain downloadable but show a deprecation warning at install time.

## Rollback plan

`npm dist-tag add pixelcheck@1.1.2 latest` repoints `latest` if any post-publish issue surfaces. No database migrations in this release.
