# Translation review template

> **Purpose**: Collect native-speaker feedback on the 90 user-facing strings
> shipped in `src/core/i18n.ts` for each non-English locale. We ship v1.0
> with machine-assisted translations + native review pass; this template is
> what reviewers fill in.
>
> **Reviewer instructions**: Skim the table below. For each row, leave the
> "Reviewer feedback" cell **blank** if the translation reads naturally to
> a native speaker in the report context. Write a suggested replacement
> only when the current value is wrong, awkward, or culturally off.

## How to use this template

1. **Pick a locale** — `zh-CN` / `ja` / `es` / `de`. (English is the source.)
2. **Copy this file** to `docs/translation-review-<locale>-<reviewer>.md` (e.g. `docs/translation-review-zh-CN-2026-05-jane.md`).
3. **Fill the metadata block** at the top with reviewer name, date, locale.
4. **For each row**, leave "Suggested replacement" blank if the current value is fine. Write only the replacement string when changing — no "looks good" / "ok" comments needed.
5. **Submit a PR** with the filled template + corresponding edits to the
   relevant locale dictionary in `src/core/i18n.ts`.

Each locale has **90 keys** mirroring the English source. The
`lintTranslations()` unit test enforces 100% key coverage, so the
dictionary cannot be incomplete — only quality-reviewed or not.

## Cross-cutting feedback

Beyond per-key review, please answer:

### Style & register

- Is the overall register (formal / casual / technical) consistent across the dictionary?
- Are there phrases that read like literal translations rather than native idiom?
- Are issue severity terms (`critical` / `high` / `medium` / `low`) used naturally in your locale's QA / audit vocabulary, or do they need different terms?

### Cultural / contextual fit

- Does the term "audit" carry connotations in your locale that might surprise a non-native reader of the report? (Some locales reserve "audit" for financial/legal contexts.)
- Is "AI Browser Auditor" as a product name acceptable transliterated, or should we localise the product name itself?
- Are there any strings that need region-specific variants (e.g., `zh-CN` vs `zh-TW` differ for some terms)?

### Number / unit / date format

- Currency: USD is shown as the cost unit. Is this the right convention for your locale's reports, or should we localise (`美元` / `米ドル` / `dólares EE.UU.` / `US-Dollar`)?
- Date format: `Run date` precedes a timestamp formatted by `Intl.DateTimeFormat`. Any locale-specific gotchas to flag?

### Missing keys

Are there concepts the report needs that are awkward to express with the current key set? (We can add keys; this drives v1.x dictionary expansion.)

## How this is integrated back

After PR merge:

1. Maintainer applies suggested replacements to `src/core/i18n.ts`
2. CHANGELOG entry: `### Changed — Translations: <locale> reviewed by <reviewer> (XX corrections applied)`
3. README "Translations reviewed by:" section adds the reviewer
4. Filled review template stays in `docs/translation-review-<locale>-<reviewer>.md` for future reference

We track reviewers publicly because translation review is professional
work and recognition matters.
